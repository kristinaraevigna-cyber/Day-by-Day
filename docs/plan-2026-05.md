## Reframing against what's actually there

The earlier plan treated this as a greenfield build. It isn't. The real app has:

- 8 written chapters (industry-neutral content, one long page each)
- 8 implemented assessment instruments (no charts)
- Two working AI coaches (ICF + Day Advisor), voice and text, with action capture
- Journal, Actions, Practice activities (VCoL), Reading Library
- Supabase auth + consent gate
- Render deployment with autodeploy on `main` push (to confirm)

What it does **not** have, and what the prompt either misdescribes or hasn't built yet:

- **No global pre/mid/post phase system.** The `currentWeek=1` bug in the prompt doesn't exist — there's no surrounding system at all. Build, don't fix.
- **No staged module architecture.** Chapters are one long page each; the 5-stage (Frame/Assess/Interpret/Practice/Integrate) split is the redesign.
- **No AWS instrument.** The prompt names AWS as the demo centerpiece — it has to be added (28 items, six subscales, plus the radar chart that's also missing).
- **No healthcare framing.** Content is industry-neutral. Reframing means *layering on* healthcare context, not replacing first-responder content.
- **No role selection in onboarding.** Phase 4 onboarding redesign is real.
- **No push-survey / banner / toast primitive.** Phase 3 infra is greenfield.
- **No schema files in repo.** We're inferring schema from client code. Phase 1 should commit a `supabase db pull` snapshot.
- **No tests, no `.env.example`.** Worth fixing in Phase 1 housekeeping.

The redesign is meaningfully smaller than a greenfield build, but the *demo-critical net-new pieces* (AWS, radar chart, pre/mid/post phasing, demo-mode toggle, staged module rewrite, push surveys, banners, role-based onboarding) are still ~80% of the work the prompt describes.

## Working-on-main risk note

You said: "working directly on main, simpler workflow, no live users to protect." Acknowledged — committing directly. Two safety practices I'll observe anyway:

1. **Each commit compiles and runs.** Per the prompt's working agreement. If a commit breaks the build, the next push breaks Render. With no live users that's recoverable, but it still costs a demo-rehearsal window.
2. **Push at meaningful checkpoints, not every commit.** I'll commit incrementally locally and push when a checkpoint is verifiable (e.g., "stage 1 of module 1 renders end-to-end," not "wired up a button"). Reduces noise and keeps the Render deployment history readable.

If at any point you want me to switch to a feature branch for a risky chunk (say, the assessment phase logic), say the word.

## Path B — "Demo build" (one week) — RECOMMENDED

The point of the demo is for Day to walk a CHRO through a credible architecture, not to ship a finished product. Path B builds the architecture and the first 3 modules well, stubs the rest visibly, and ships AWS + radar chart as the proof point.

**Realistic time:** 35–45 hours. Tight but achievable.

### What gets fully built

- **Pre-step items** (3 h) — schema pull, env, Tailwind sanity, `.env.example`.
- **Staged module architecture** (8 h) — `ChapterDetail` refactor: each chapter renders as 5 stages (Frame / Assess / Interpret / Practice / Integrate). Action-gated unlock by default. Apply to **chapters 1, 2, 3 only**. Chapters 4–8 stay one-page and get a "Staged delivery coming Week N" banner inside.
- **Pre/mid/post phase system** (5 h) — `enrolled_at` on `user_profiles` (set on first login or on consent acceptance). Date math against `MIDPOINT_WEEK=4` and `POST_WEEK=8`. `duePhase` derived state. Baseline gate redirects to baseline assessment on app load if not done.
- **AWS implementation** (4 h) — add 28 items in six subscales, scoring, results screen with **radar chart** (using `recharts`, one new dep). This is the demo centerpiece.
- **Demo-mode admin toggle** (2 h) — button visible only when user's email is on an admin allowlist; flips current user's phase between baseline/midpoint/post in one click. Hidden behind a settings menu in production, surfaced for the demo account.
- **Secondary instruments — stubbed** (2 h) — show instrument name, item count, one-line evidence statement, "Coming in production" badge. No scoring. Includes Copenhagen Burnout, PERMA-Profiler (note: distinct from existing PERMA+4), BRS, SRIS, ALQ-16, WAI-SR, SUS.
- **Post-session ratings** (3 h) — modal after every coach session (ICF or Day, voice or text). 3 Likert items. `session_ratings` table. Wired into both `VoiceCoach` (on disconnect) and `TextCoach` (on session end).
- **Push-survey primitive + dashboard banner** (4 h) — generic `<Banner />` component with priority queue ("Baseline due" > "Midpoint ready" > "Unfinished session" > "Module reflection due"). Drives post-session ratings + assessment due notices. Weekly pulse trigger logic stubbed.
- **Module-completion reflection** (1 h) — short modal on final stage of a chapter.
- **Healthcare reframing — chapters 1–3 + coach prompts** (5 h) — rewrite chapter copy, practice examples for chapters 1–3 only with healthcare scenarios (handovers, escalation, MDTs, staffing, moral distress, second-victim, just culture). Update both coach system prompts (ICF voice + text, Day Advisor voice + text) with healthcare terminology + guardrails (no clinical/HR/legal advice; EAP / occupational health / peer-support redirect for crisis or impaired-colleague concerns). Chapters 4–8 stay industry-neutral and labeled "Staged delivery coming Week N."
- **Role-based onboarding** (3 h) — after signup + consent: role select (Nurse Manager / Charge Nurse / Nursing Director / CNO / Physician Leader / Medical Director / CMO / Hospital Administrator / Other Healthcare Leader / Non-healthcare) → "Do you have direct reports?" (yes / no / informal) → baseline assessment. Store in `user_profiles`. Coach system prompts read role and prime examples accordingly.
- **Dashboard simplification** (2 h) — collapse to 3 primary cards (Today's stage, Talk to your coach, Latest reflection). Move streaks/practice/library to a Progress tab. Every card has an estimated time ("3 min", "7 min").
- **Evidence-base screen** (1 h) — static page listing every instrument used, with brief citation. Linked from settings.
- **Sample research output mock** (1 h) — static screen showing illustrative aggregate radar deltas + effect sizes. Labeled "Illustrative — not real data." For the CHRO.
- **Demo users** (1 h) — 3–5 seeded users at different roles and phases. Documented in `docs/demo-users.md`.

- Six secondary instruments: name + items + citation, no scoring.
- Weekly pulse survey: trigger logic in place, payload empty.

- Researcher dashboard: not in scope; sample-output mock substitutes.

## Recommendation

**Path B.** Reasons:
1. Staged module architecture is the most novel thing about the redesign and is the easiest piece to demo. Cutting it makes the demo feel like "the existing app with a few new features."
2. Path B's risk surface is manageable now that the app already exists and we're not building auth/routing/coaches/journal from scratch.
3. The AWS + radar + demo toggle combination is the high-leverage demo move; Path B has time to land it well.

Cut to Path C if:
- Tailwind triage in the pre-step turns into a 4-hour rabbit hole, OR
- The Supabase schema pull reveals schema-drift problems that need a day of cleanup, OR
- Day asks for fewer-but-more-polished screens.

Cut to Path A only as the post-demo roadmap.
